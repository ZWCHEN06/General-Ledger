# 个人记账 App 开发规则

本文档用于约束个人记账 App 第一版的开发范围、技术选型和协作流程。所有开发任务都应优先遵守本文档，避免第一版范围失控。

## 1. 项目定位

本项目是一个本地优先的个人记账 App。

第一版目标：

- 用户可以在手机本地记录收入和支出。
- 用户可以查看本月收入、本月支出和本月结余。
- 用户可以查看、编辑、删除历史记录。
- 所有界面文案使用中文。

第一版不追求完整财务系统能力，只完成稳定、清晰、可维护的本地记账 MVP。

## 2. 技术栈

第一版固定使用以下技术栈：

- Expo React Native
- TypeScript
- SQLite

技术约束：

- 使用 Expo 管理项目运行和构建流程。
- 所有业务代码使用 TypeScript。
- 本地数据持久化使用 SQLite。
- 不引入复杂状态管理库。
- 不为了第一版引入大型 UI 框架或复杂基础设施。

## 3. 第一版功能范围

第一版只做以下功能：

- 添加收入记录。
- 添加支出记录。
- 每条记录包含：
  - 类型
  - 金额
  - 分类
  - 日期
  - 备注
- 查看本月收入。
- 查看本月支出。
- 查看本月结余。
- 查看历史记录。
- 编辑历史记录。
- 删除历史记录。

## 4. 第一版明确不做

以下功能第一版禁止开发：

- 登录
- 注册
- 云同步
- 支付功能
- 复杂动画
- 多设备同步
- 多人共享账本
- 服务端接口
- 远程数据库
- 复杂状态管理库
- 预算管理
- 图表分析
- 账单导入
- 图片或发票上传
- 多币种
- 多账户管理

如确实需要新增超出范围的功能，必须先更新本文档或新增需求文档，再开始开发。

## 5. 数据规则

第一版数据只保存在本地 SQLite。

核心数据表建议为 `records`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER | 主键，自增 |
| `type` | TEXT | 记录类型：`income` 或 `expense` |
| `amount` | INTEGER | 金额，单位为分 |
| `category` | TEXT | 分类名称 |
| `date` | TEXT | 记账日期，建议使用 `YYYY-MM-DD` |
| `note` | TEXT | 备注，可为空 |
| `created_at` | TEXT | 创建时间 |
| `updated_at` | TEXT | 更新时间 |

数据实现规则：

- 金额统一用整数分保存，避免浮点数精度问题。
- 日期统一存储为字符串，格式为 `YYYY-MM-DD`。
- 收入和支出使用同一张记录表，通过 `type` 区分。
- 删除记录第一版可以使用物理删除。
- 第一版分类可以先使用固定选项，不必开发分类管理功能。

## 6. 状态管理规则

第一版不引入 Redux、MobX、Zustand 等复杂状态管理库。

优先使用：

- React `useState`
- React `useEffect`
- React Context，仅在确实需要跨页面共享状态时使用
- 本地 SQLite 查询结果作为数据来源

状态管理原则：

- 页面内状态放在页面组件内。
- 可复用逻辑抽到自定义 Hook。
- 数据读写逻辑与 UI 组件分离。
- 不为少量状态提前设计复杂架构。

## 7. UI 与交互规则

界面规则：

- 所有界面文案使用中文。
- 页面结构优先清晰、直接、易操作。
- 不做复杂动画。
- 不为了视觉效果牺牲记账效率。
- 表单字段必须有清晰标签。
- 金额输入必须限制为合法数字。
- 删除记录前需要二次确认。

基础页面建议：

- 首页：展示本月收入、本月支出、本月结余，并提供添加记录入口。
- 添加记录页：新增收入或支出。
- 历史记录页：查看历史记录，并支持进入编辑或删除。
- 编辑记录页：修改已有记录。

## 8. 开发流程规则

每次开发必须遵守以下流程：

1. 每次只完成一个小功能。
2. 修改代码前，先说明准备修改哪些文件。
3. 修改时只改与当前功能相关的文件。
4. 修改完成后，必须给出测试步骤。
5. 测试步骤应包含手动验证路径和预期结果。
6. 如果发现需求不清晰，先补充需求，再写代码。
7. 不做顺手重构，除非它是完成当前功能所必需的。

小功能示例：

- 创建 SQLite 数据表。
- 实现添加支出记录。
- 实现添加收入记录。
- 实现本月统计查询。
- 实现历史记录列表。
- 实现编辑记录。
- 实现删除记录。

## 9. 代码组织规则

建议按职责拆分代码：

- `screens/`：页面组件。
- `components/`：可复用 UI 组件。
- `db/`：SQLite 初始化、迁移、查询方法。
- `types/`：TypeScript 类型定义。
- `utils/`：金额、日期等通用工具方法。

代码规则：

- TypeScript 类型必须明确。
- 数据库读写方法要集中管理。
- 页面组件不直接拼接复杂 SQL。
- 金额格式化和日期格式化应封装为工具函数。
- 命名应清晰表达业务含义。

## 10. 测试与验收规则

每个功能完成后至少进行手动测试。

测试步骤应说明：

- 打开哪个页面。
- 执行什么操作。
- 输入什么数据。
- 预期页面显示什么结果。
- 预期 SQLite 数据如何变化。

每个功能的最低验收标准：

- App 能正常启动。
- 当前功能主流程可用。
- 不破坏已有功能。
- 控制台没有明显运行错误。
- 中文文案显示正常。

## 11. 后续扩展方向

以下功能可以作为后续版本扩展，但不进入第一版：

- 自定义分类。
- 月度统计。
- 分类统计。
- 图表分析。
- 搜索和筛选。
- 预算设置。
- 数据导出。
- 数据备份与恢复。
- 登录。
- 云同步。
- 多设备同步。
- 多账户管理。
