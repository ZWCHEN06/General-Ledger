# 个人记账 App 开发规则

本文档用于约束个人记账 App 第一版的开发范围、技术栈、职责边界和协作流程。所有开发任务都应优先遵守本文档，避免第一版范围失控。

## 1. 项目定位

本项目是一个本地优先的个人记账 App。

第一版只做桌面版，不直接做 Android。Android 适配只能作为后续版本规划，必须在桌面版核心流程稳定后再评估。

第一版目标：

- 用户可以在本地记录收入和支出。
- 用户可以查看本月收入、本月支出和本月结余。
- 用户可以查看、编辑、删除历史记录。
- 所有界面文案使用中文。
- 数据默认只保存在本机。

第一版不追求完整财务系统能力，只完成稳定、清晰、可维护的本地记账 MVP。

## 2. 技术栈

第一版固定使用以下技术栈：

- Qt 6
- QML
- C++
- SQLite
- CMake

技术约束：

- 使用 CMake 管理构建、资源和目标配置。
- 使用 Qt 6 构建桌面应用。
- 使用 QML 编写界面。
- 使用 C++ 实现数据模型、数据库访问、业务逻辑和统计计算。
- 使用 SQLite 作为本地持久化存储。
- 不为了第一版引入服务端、账号系统或复杂基础设施。

## 3. 职责边界

### QML 职责

QML 只负责界面和轻量交互：

- 页面布局。
- 表单展示。
- 按钮、输入框、列表等 UI 控件。
- 页面跳转和简单交互状态。
- 调用 C++ 暴露的方法或模型。

QML 禁止承担以下职责：

- 不直接拼接或执行 SQL。
- 不实现核心业务规则。
- 不实现统计计算。
- 不直接管理 SQLite 连接。
- 不保存复杂应用状态。
- 不写大量 JavaScript 业务逻辑。

### C++ 职责

C++ 负责应用核心能力：

- 数据模型。
- SQLite 初始化、迁移、查询和写入。
- 收入、支出、分类、日期、金额等业务规则。
- 本月收入、本月支出、本月结余等统计计算。
- 向 QML 暴露可用的 QObject、QAbstractListModel 或服务类。
- 输入数据校验。
- 错误处理和日志输出。

### SQLite 职责

SQLite 只负责本地存储：

- 保存记账记录。
- 保存第一版必要的基础配置。
- 支持本地查询和统计。

第一版不使用远程数据库，不做云端备份，不做多设备同步。

## 4. 第一版功能范围

第一版只做以下功能：

- 添加收入记录。
- 添加支出记录。
- 每条记录包含：
  - 类型
  - 金额
  - 分类
  - 日期
  - 备注
- 查看本月收入。
- 查看本月支出。
- 查看本月结余。
- 查看历史记录。
- 编辑历史记录。
- 删除历史记录。

每次开发只能完成一个小功能，不把多个功能合并到一次修改中。

## 5. 第一版明确不做

以下功能第一版禁止开发：

- 登录
- 注册
- 云同步
- 支付功能
- 复杂动画
- Android 版本
- iOS 版本
- 多设备同步
- 多人共享账本
- 服务端接口
- 远程数据库
- 复杂权限系统
- 复杂状态管理框架
- 账单导入
- 图片或发票上传
- 多币种
- 多账户管理

如确实需要新增超出范围的功能，必须先更新本文档或新增需求文档，再开始开发。

## 6. 数据规则

第一版数据只保存在本地 SQLite。

核心数据表建议为 `records`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER | 主键，自增 |
| `type` | TEXT | 记录类型：`income` 或 `expense` |
| `amount` | INTEGER | 金额，单位为分 |
| `category` | TEXT | 分类名称 |
| `date` | TEXT | 记账日期，建议使用 `YYYY-MM-DD` |
| `note` | TEXT | 备注，可为空 |
| `created_at` | TEXT | 创建时间 |
| `updated_at` | TEXT | 更新时间 |

数据实现规则：

- 金额统一用整数分保存，避免浮点数精度问题。
- 日期统一存储为字符串，格式为 `YYYY-MM-DD`。
- 收入和支出使用同一张记录表，通过 `type` 区分。
- 删除记录第一版可以使用物理删除。
- 第一版分类可以先使用固定选项，不必开发分类管理功能。
- 数据库读写必须通过 C++ 封装，QML 不直接访问 SQLite。

## 7. 代码组织规则

建议按职责拆分代码：

- `CMakeLists.txt`：项目构建配置、Qt 模块、资源文件和可执行目标。
- `main.cpp`：应用入口、QML 引擎初始化、C++ 类型注册。
- `Main.qml`：主界面入口。
- `src/db/`：SQLite 连接、初始化、迁移、查询封装。
- `src/models/`：暴露给 QML 的数据模型。
- `src/services/`：记账记录业务逻辑、校验和写入流程。
- `src/statistics/`：收入、支出、结余等统计计算。
- `src/types/`：C++ 数据结构、枚举和公共类型。
- `qml/`：QML 页面和可复用界面组件。

代码规则：

- C++ 类命名应清晰表达业务含义。
- 数据库读写方法要集中管理。
- QML 页面不直接拼接 SQL。
- 金额格式化和日期格式化应封装为 C++ 或明确的 UI 辅助方法。
- C++ 暴露给 QML 的接口应保持简单、稳定。
- 不做顺手重构，除非它是完成当前小功能所必需的。

## 8. UI 与交互规则

界面规则：

- 所有界面文案使用中文。
- 页面结构优先清晰、直接、易操作。
- 第一版不做复杂动画。
- 不为了视觉效果牺牲记账效率。
- 表单字段必须有清晰标签。
- 金额输入必须限制为合法数字。
- 删除记录前需要二次确认。
- 桌面版优先适配常见窗口尺寸，不先处理 Android 触控适配。

基础页面建议：

- 首页：展示本月收入、本月支出、本月结余，并提供添加记录入口。
- 添加记录页：新增收入或支出。
- 历史记录页：查看历史记录，并支持进入编辑或删除。
- 编辑记录页：修改已有记录。

## 9. 开发流程规则

每次开发必须遵守以下流程：

1. 每次只完成一个小功能。
2. 修改前先说明准备修改哪些文件。
3. 修改时只改与当前功能相关的文件。
4. 修改完成后必须给出测试步骤。
5. 测试步骤应包含手动验证路径和预期结果。
6. 如果发现需求不清晰，先补充需求，再写代码。
7. 不做顺手重构，除非它是完成当前功能所必需的。

小功能示例：

- 配置 Qt 6 + CMake 基础工程。
- 创建 SQLite 数据库连接。
- 创建 `records` 数据表。
- 实现添加支出记录。
- 实现添加收入记录。
- 实现本月统计查询。
- 实现历史记录列表模型。
- 实现编辑记录。
- 实现删除记录。

每次修改前说明文件时，应使用类似格式：

```text
本次准备修改：
- CMakeLists.txt：增加 Qt Sql 模块。
- src/db/DatabaseManager.h：新增数据库管理类声明。
- src/db/DatabaseManager.cpp：实现 SQLite 初始化。
```

每次修改后说明测试步骤时，应使用类似格式：

```text
测试步骤：
1. 使用 CMake 构建并启动桌面 App。
2. 打开添加记录页面。
3. 输入一条支出记录并保存。
4. 回到首页，确认本月支出增加。
5. 打开历史记录页，确认新记录显示正常。
```

## 10. 测试与验收规则

每个功能完成后至少进行手动测试。涉及 C++ 业务逻辑或统计逻辑时，优先补充轻量级单元测试或可重复验证步骤。

测试步骤应说明：

- 如何构建项目。
- 如何启动桌面 App。
- 打开哪个页面。
- 执行什么操作。
- 输入什么数据。
- 预期页面显示什么结果。
- 预期 SQLite 数据如何变化。

每个功能的最低验收标准：

- App 能正常构建。
- App 能正常启动。
- 当前功能主流程可用。
- 不破坏已有功能。
- 控制台没有明显运行错误。
- 中文文案显示正常。
- 数据保存和读取结果符合预期。
